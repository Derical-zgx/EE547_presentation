02 / Overview

Project Summary

  • Problem: esports statistics are scattered across websites and hard to analyze.
  • FPS-MetaSync centralizes Valorant match, map, round, and player statistics.
  • Admins trigger async sync jobs; users browse matches, teams, players, and archetypes.
  • ML component uses K-means to group players by play style.
  • Goal: build a cloud-backed esports analytics pipeline with a usable frontend.

Current status

  • Implemented schema, API, worker, scraper, ML pipeline, SPA UI, Docker docs
  • Initialized RDS schema and admin account
  • Partial full data refresh validation and EC2 production run
03 / Architecture

Architecture Summary

Web Tier

  • Frontend: vanilla JavaScript SPA served by Flask.
  • Backend API: Flask + SQLAlchemy with JWT-protected admin routes.

Data Tier

  • Database: PostgreSQL on AWS RDS.
  • Normalized tables plus raw JSONB payloads.

Async Tier

  • Queue: AWS SQS decouples admin actions from slow jobs.
  • Worker: scrapes VLR.gg, upserts data, runs K-means.
Implemented vs. stubbed: Valorant/VLR path implemented; CS2 and scheduled refresh are planned.

05 / Context

Background

  • Valorant esports data includes series, maps, rounds, and player statistics.
  • Useful player signals include rating, ACS, ADR, K/D, KAST, FK/FD, role, and agent pool.
  • VLR.gg provides rich match pages, but not a ready-to-use analytics dashboard.
  • Batch ingestion is needed because scraping is slow and should not block web requests.
  • Unsupervised clustering helps identify player styles without official labels.

Backend

17 / Backend

Backend Summary

  • Language/frameworks: Python, Flask, SQLAlchemy 2.x, boto3, psycopg2.
  • Routing stack: API endpoints in api/app.py; shared models in shared/models.py.
  • Database: PostgreSQL on AWS RDS with normalized tables and raw JSONB payloads.
  • Async work: API sends SQS messages; worker long-polls and dispatches jobs.

  • Layout: api/, worker/, shared/, frontend/, db/, docker/, docs/.

19 / Backend

Outside Data Source / API - VLR.gg via vlrdevapi

  • Source: VLR.gg public Valorant match pages and vlrdevapi Python library.
  • Access pattern: worker pulls completed match IDs and fetches full match details.
  • Stored data: series metadata, teams, maps, scores, rounds, and player stats.
  • Filtering: current sync pulls recent completed matches with configurable limits.
  • Rate control: retry/backoff handles rate limits, network errors, and scraping errors.
  • Data is written idempotently, so repeated jobs do not duplicate records.
20 / Backend

Performance and Scalability

  • Slow operations are asynchronous, so scraping and K-means do not block HTTP requests.
  • SQS provides buffering, retry behavior, and room for horizontal worker scaling.
  • Database indexes support common match, team, player, cluster, and stat queries.
  • Upserts make sync safe to rerun after partial failures.
  • Known bottleneck: VLR.gg rate limits, so the worker sleeps between match fetches.
  • Future scaling: scheduled sync, batch tuning, RDS pool tuning, and API caching.
21 / Backend

DevOps / System Operations

  • CI/CD: not implemented yet; repo is organized for future GitHub Actions.

Section - 03

Machine Learning

Unsupervised player-style classification with K-means.

23 / ML

ML Overview

  • Model type: custom unsupervised K-means clustering, default k = 4.
  • Purpose: group players into interpretable play-style archetypes.
  • Inputs: aggregated player performance features from player_map_stats.
  • Output: cluster_id and human-readable archetype label stored on players, shown in UI.
  • Framework: scikit-learn StandardScaler + KMeans; numpy fallback if sklearn unavailable.

Archetype examples

  • Entry Fragger
  • Lurker / AWPer
  • Support Initiator
  • Anchor IGL
  • Flex Star
24 / ML

ML Integration

Same async pattern as the data sync (Slide 18) — admin click triggers an SQS job; the pipeline runs in the worker.

Trigger flow

  • Admin clicks Train K-means in the SPA.
  • API — POST /api/admin/train → SQS message {type: train_kmeans, k, min_matches} → HTTP 202.
  • Worker dispatches to worker/ml_pipeline.py::train_and_persist().
  • No external scraping — the pipeline reads only what is already in RDS.

Writes back to RDS

  • player_features — one snapshot row per player per run
  • ml_clusters — centroid (JSONB), size, silhouette, run_id
  • players.ml_cluster_id + ml_archetype + ml_scored_at

Trained model = data: no .pkl file. Centroids live in ml_clusters.centroid JSONB; each run gets a new run_id, so history is preserved.

25 / ML

ML Pipeline — 4 Steps from Stats to Archetype

Step ①

SQL Aggregate

  • player_map_stats → 17 features per player
  • Combat, KAST, FK rate, agent_flex, 4 role pcts
  • HAVING n_maps >= :min_matches filter
single CTE in PostgreSQL
Step ②

Preprocess

  • Fill NaN with column mean
  • z-score standardize using StandardScaler
  • Avoids ACS (~200) dominating KAST (0–1)
numpy / sklearn
Step ③

K-means Clustering

  • KMeans(n_clusters=k, n_init=10)
  • Default k = 4, parameterized per request
  • Pure-numpy fallback if sklearn missing
silhouette_score (eval)
Step ④

Centroid Template Scoring

  • 5 archetype templates — weight vectors
  • score = template · centroid_norm
  • Greedy → unique labels
interpretable, no black box
Persisted player_features · ml_clusters · players.ml_archetype Labels Entry Fragger Lurker / AWPer Support Initiator Anchor IGL Flex Star
26 / ML

ML Performance

  • Metrics implemented: silhouette_score when sklearn and enough samples are available.
  • Latency expectation: seconds for small demo datasets; longer runs stay as batch jobs.
  • Interpretability: centroid templates turn feature patterns into readable labels.
  • Known limitations: no ground truth; labels are heuristic and need domain review.
  • Potential failure modes: too few players, VLR field changes, unbalanced clusters, rate limits.
  • Future work: compare k=2..8, add PCA plots, store scaler metadata, evaluate stability.

29 / API

API Security and Access

  • Authentication: /auth/login validates credentials and returns a signed JWT.
  • Authorization: admin routes require token and admin role.
  • Protected endpoints: POST /api/admin/sync and POST /api/admin/train.
  • Public read endpoints: matches, teams, players, and clusters for viewer browsing.
  • Server-side verification: jwt.decode with SECRET_KEY, then user lookup in PostgreSQL.
  • Production recommendation: rotate demo secrets, remove fallback DB URL, and use EC2 IAM role.
