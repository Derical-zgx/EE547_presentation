02 / Overview

Project Summary

  • Problem. Esports statistics are scattered across websites and hard to turn into team / player insights.
  • FPS-MetaSync centralizes Valorant match, map, round, and player statistics into a queryable web system.
  • Admins trigger asynchronous data sync jobs; users browse matches, teams, player cards, and ML archetypes.
  • ML component groups players by play style using K-means over recent performance features.
  • Goal. Demonstrate a cloud-backed data pipeline plus a usable analytics frontend for an esports domain.

Current status

  • Done Schema, API, worker, scraper, ML, SPA, Docker docs
  • Done RDS schema and admin account initialized
  • Partial Full data refresh validation & EC2 production run
03 / Architecture

Architecture Summary

Web Tier

  • Frontend. Vanilla JavaScript SPA served by Flask.
  • Backend API. Flask + SQLAlchemy, JWT auth, REST endpoints for matches / teams / players / clusters.
  • Database. PostgreSQL on AWS RDS — normalized tables + JSONB raw payloads.

Async Tier

  • Queue. AWS SQS decouples admin requests from slow scraping / ML training jobs.
  • Worker. Consumes SQS, calls vlrdevapi, upserts data, runs batch K-means.
  • Scope. Valorant / VLR path implemented; CS2 support and scheduled refresh are planned.

05 / Context

Background

  • Valorant esports matches contain series-level, map-level, round-level, and per-player statistics.
  • Useful play-style signals: ACS, rating, K/D/A, ADR, KAST, first kills / deaths, role, agent pool.
  • Public sites such as VLR.gg expose rich pages, but they are not designed for programmatic analytics dashboards.
  • Batch ingestion is appropriate — scraping can be slow, rate-limited, and should not block web requests.
  • Unsupervised clustering is useful when no official label exists for play-style categories.

Section · 02

Backend

REST API, database schema, queue worker, and data ingestion.

16 / Backend

Backend Summary

  • Stack: Python · Flask · Flask-SQLAlchemy / SQLAlchemy 2.x · boto3 · psycopg2-binary.
  • Routing: API endpoints in api/app.py; shared SQLAlchemy models in shared/models.py.
  • Database: PostgreSQL on AWS RDS — normalized relational tables + raw JSONB payloads.
  • Async: API → SQS messages; worker/worker.py long-polls and dispatches handlers.
  • Persistence: worker/data_writer.py uses PostgreSQL ON CONFLICT upserts → idempotent sync.
  • Layout: api/, worker/, shared/, frontend/, db/, docker/, docs/.

Tables (10)

  • users
  • teams, players
  • matches, maps, rounds
  • player_map_stats
  • player_features, ml_clusters
17 / Backend

Outside Data Source — VLR.gg via vlrdevapi

  • Source. VLR.gg public Valorant match pages and the third-party vlrdevapi Python library.
  • Access pattern. Worker pulls completed match IDs, then fetches full series detail per match.
  • Stored data. Series metadata, two teams, map veto, map scores, round outcomes, 10 player stat lines per map.
  • Filtering. Sync pulls most recent completed matches with configurable limit / pages.
  • Rate control. Retry wrapper handles RateLimitError, NetworkError, ScrapingError with sleeps / backoff.
  • Idempotent. Repeated jobs do not duplicate matches or player-map stats.
18 / Backend

Performance & Scalability

  • Slow operations are asynchronous — scraping & K-means do not block HTTP requests.
  • SQS gives backpressure and retry behavior; worker can be scaled horizontally for more throughput.
  • DB indexes on match_date, team IDs, player IDs, cluster IDs, and map / stat foreign keys.
  • Upserts make sync safe to re-run after failures or partial fetches.
  • Known bottleneck. VLR.gg scraping rate limits — worker intentionally sleeps between fetches.
  • Future scaling. Scheduled sync, batch size tuning, RDS connection pool tuning, response cache.
19 / Backend

DevOps / System Operations

  • CI/CD. Not yet implemented; repository is organized for GitHub Actions later.
  • Artifacts. Dockerfile.api, Dockerfile.worker, docker-compose.yml, DEPLOY.md.
  • Config. .env.example — DATABASE_URL, SQS_QUEUE_URL, AWS_REGION, SECRET_KEY, optional AWS keys.
  • Monitoring. Stdout / stderr from Flask / gunicorn and worker logging.
  • Errors. Worker catches task exceptions, rolls back DB transactions, leaves failed SQS messages for retry.
  • Lessons. IAM SQS permissions and RDS security-group / network settings were the key integration risks.
Section · 03

Machine Learning

Unsupervised player-style classification with K-means.

21 / ML

ML Overview

  • Model. Custom-trained unsupervised K-means clustering, default k = 4.
  • Purpose. Group players into interpretable play-style archetypes for the frontend.
  • Inputs. Aggregated player performance features from player_map_stats.
  • Output. cluster_id and human-readable archetype label stored on players, shown in UI.
  • Framework. scikit-learn StandardScaler + KMeans; numpy fallback if sklearn unavailable.

Archetype examples

  • Entry Fragger
  • Lurker / AWPer
  • Support Initiator
  • Anchor IGL
  • Flex Star
22 / ML

ML Integration

  • Inference / training in worker/ml_pipeline.py as an async batch job.
  • Admin clicks Train K-means → Flask API sends {type: train_kmeans} to SQS.
  • Worker loads features directly from PostgreSQL using aggregate SQL.
  • Pipeline inserts player_features snapshots and ml_clusters rows for the run.
  • Pipeline updates players.ml_cluster_id, players.ml_archetype, players.ml_scored_at.
  • Async design avoids slow ML work inside request handlers — UI stays responsive.
23 / ML

ML Data & Training

  • Source. Recent per-map player stat lines scraped from VLR.gg.
  • Features. Combat (ACS, rating, ADR, kills/round) · Survivability (KAST, deaths/round) · Entry impact (FK / FD) · Role / agent flexibility.
  • Filter. Players must have ≥ min_matches maps to avoid unstable clusters.
  • Preprocessing. Missing values filled with column means; features standardized before K-means.
  • Storage. No serialized model file — centroids and labels are stored in ml_clusters JSONB.
  • Limitation. Cluster quality depends on the amount of scraped match history available in RDS.
24 / ML

ML Performance

  • Metrics. silhouette_score when sklearn is available and enough samples exist.
  • Latency. Seconds for class-demo datasets; longer with many players, but suitable as batch work.
  • Interpretability. Centroid templates label clusters by high / low feature patterns rather than opaque numbers.
  • Limitations. No supervised ground truth; labels are heuristic and should be validated by domain review.
  • Failure modes. Too few player rows · VLR field changes · unbalanced clusters · rate-limited collection.
  • Future work. Compare k = 2..8, add PCA visualization, persist scaler / model metadata, evaluate cluster stability.

What was built, what works, and what remains.

30 / Wrap

Anything Else

  • Hybrid storage. The DB stores both normalized query-friendly rows and raw JSONB payloads for audit / debug.
  • Idempotent ingestion. External IDs from VLR (team_id, player_id, match_id, game_id) are primary / unique keys.
  • Domain-specific labeling. Archetypes are assigned by centroid templates over esports-specific features.
  • Engineering tradeoff. API uses Python 3.11 container; worker uses Python 3.13 for vlrdevapi compatibility.
  • Course fit. Combines cloud infrastructure, async queues, database design, backend APIs, frontend UI, and ML pipeline.
